                                                                            Mysql在可重复读隔离级别下如何保证事务较高的隔离性，同样的sql查询语句在一个事务里多次执行查询结果相同，就算其他事务对数据有修改也不会影响当前事务sql语句的查询结果。

                                                                            这个隔离性在Mysql中就是靠MVCC（Multi-Version Concurrency Control）机制保证的，对一行数据的读和写两个操作是不会通过加锁互斥来保证隔离性，避免了频繁加锁互斥，而在串行化隔离界别为了保证较高隔离性通过加锁互斥来实现。

                                                                            Mysql在读已提交和可重复读隔离级别下都实现了MVCC。

                                                                            # undo日志版本链和read view机制

                                                                            # 原理

                                                                            undo日志版本连是指一行数据被多个事务依次修改后，在每个事务修改完成后，mysql会保留修改前的数据undo回滚日志，并且用两个隐藏的字段trx_id和roll_pointer把undo日志串联起来形成一个历史记录版本链。

                                                                            在可重复读隔离级别，当事务开启后，执行任何查询sql时会生成当前事务的一致性视图（read view），该视图在事务结束前都不会有任何变化（如果是读已提交的隔离级别在每次执行查询sql时都会重新生成），这个视图由执行查询时所有未提交事务id数组（数组里最小的id为min_id）和已经创建的最大事务id（max_id）组成，事务里任何sql查询结果需要对应版本链里的最新数据开始逐个跟read view作对比从而获得最终的快照版本。

                                                                            # 版本链对比规则：

                                                                            1. 如果 row 的 trx_id 落在绿色部分（trx_id<min_id），表示这个版本是已经提交的事务生成的，这个数据时可见的。
                                                                            2. 如果 row 的 trx_id 落在红色部分（trx_id>max_id），表示这个版本是由将来启动的事务生成，是不可见的（若 row 的 trx_id 就是当前自己的事务则是可见的）。
                                                                            3. 如果 row 的 trx_id 落在黄色部分（min_id<=trx_id<=max_id），那就包括两种情况：
                                                                              1. 如果 row 的 trx_id 在视图数组中，表示这个版本是由还没有提交的事务生成，不可见（自己的事务可见）。
                                                                              2. 如果 row 的 trx_id 不在视图数组中，表示这个版本是已经提交了的事务生成的，是可见的。 注意：begin/start transaction 命令并不是一个事务的起点，在执行到他们之后的第一个修改操作InnoDB表的语句，才是真正事务的启动，才会向mysql申请事务id，mysql内部是严格按照事务的启动顺序分配事务id的。

                                                                            总结：MVCC机制的实现就是通过 read-view机制和undo版本链的对比机制，使得不同的事务会根据不同的数据版本链对比读取同一条数据在版本链上的不同版本的数据。

                                                                            # 为什么不直接更新磁盘而是用BufferPool这一套复杂原理

                                                                            如果来一个请求就进行磁盘文件的随机读写，然后更新磁盘文件里的数据性能可能相当差。因为磁盘的随机读写性能是非常差的，所以直接更新磁盘文件不能让数据库抗住很高的并发。

                                                                            Mysql这套机制看起来复杂，但是可以保证每个更新请求都是更新内存Buffer Pool，然后顺序写日志文件，同事还能保证各种异常情况下数据一致性。

                                                                            更新内存的性能是极高的，然后顺序额写磁盘文件上的日志文件的性能也是非常高的，远高于随机读写磁盘文件。

                                                                            正式通过这套机制，才能让MySQL数据库在较高配置的机器上每秒抗下几千的读写请求。
