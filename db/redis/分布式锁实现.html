                                                                                    # 特点及场景

                                                                                    # 特点

                                                                                    • 键值对数据库，可以以**O(1)**的时间复杂度取出或者插入值。
                                                                                    • 存储在内存中，也可以做持久化。
                                                                                    • key的类型可以是字符串、整型、浮点，且是唯一的。
                                                                                    • value可以使String、hash、list、set、sorted set。
                                                                                    • Redis内置了复制、磁盘持久化、LUA脚本、SSL、客户端缓存、客户端代理等一系列功能
                                                                                    • 可以通过Redis sentinel和Redis Cluster进行高可用

                                                                                    # 场景

                                                                                    • 缓存：使用redis进行缓存操作
                                                                                    • 计数器：String的自增自减实现计数器，Redis读写性能非常高，很适合读写频繁的计数器设计。
                                                                                    • 分布式ID：利用自增特性，一次性请求一个很大的步长如 incr 2000，放入本地缓存，用完再请求。
                                                                                    • 海量数据统计：位图（bitmap），存储是否参与过某次活动，读过谋篇文章，是否为会员，日活统计。
                                                                                    • 会话缓存：Spring Session整合Redis进行分布式缓存设计。
                                                                                    • 分布式队列：List是个双向链表，可以实现阻塞队列。
                                                                                    • 分布式锁：可以利用SETNX实现分布式锁。
                                                                                    • 热点数据存储：最新评论、最新文章、使用list存储，ltrim取出热点数据，删除老数据。
                                                                                    • 社交类需求：set可以实现交集，实现共同好友等，Set差集实现好友推荐、文章推送。
                                                                                    • 排行榜：sorted_set实现有序性操作。
                                                                                    • 延迟队列：使用sorted_set，使用【当前时间戳+需要延长的时间】做score，消息内容作为元素，用zadd生产消息，消费者用zrange by score获取做轮询。消费完rem key number删除。

                                                                                    # Redison加锁原理

                                                                                    核心：通过setnx命令，如果能够setnx成功，说明redis中没有这个锁，线程可以操作，否则拥有锁则不能操作。

                                                                                    # Redison 加锁原理

                                                                                    1. 线程1进行加锁，操作Master进行写入。
                                                                                    2. 加锁成功，线程1启动一个后台线程，每隔一段时间访问Master看是否还持有该锁，如果持有则延长锁的时间。
                                                                                    3. 线程2进行加锁，如果加锁成功进入第一步，如果失败则自旋，直到加锁成功。

                                                                                    # Redison 源码实现

                                                                                    1. 获取当前线程id
                                                                                    2. 执行Lua脚本拿到锁：
                                                                                      1. 判断lockkey是否在redis中存在
                                                                                      2. 如果没有，使用Hash进行lock的存储，field1写入线程id，value设置为1
                                                                                      3. 用pexpire为锁设置超时时间。
                                                                                      4. 如果lockkey存在，则对应线程的value+1并且刷新时间，实现重入锁。
                                                                                      5. 如果不是本线程加锁，返回锁剩余时间。
                                                                                    3. 添加一个监听器FutureListener，开启一个定时任务TimeTask（新线程），每隔WatchDog/3的时间进行一次轮询，执行Lua脚本：
                                                                                      1. 使用hexists 判断锁是否是该线程加的
                                                                                      2. 如果是，用pexpire为锁设置超时时间
                                                                                    4. 如果没有加锁成功返回了剩余时间，则while(true)循环自旋。

                                                                                    # 带来的问题

                                                                                    由于Redis的集群架构不是强一致性的架构，如果加锁后Master挂掉，但是数据还没有同步到Slave，锁会丢失。

                                                                                    解决方法可以使用Zookeeper的分布式锁，Zookeeper是强一致性架构。还可以使用RedLock，但是不推荐使用，市面上的RedLock性能和可靠性不好，RedLock的原理是每次加锁会把锁同时加到多个节点，半数以上的节点返回成功才判定加锁成功。

                                                                                    # 高性能分布式锁

                                                                                    对于不同的锁的性能提升，可以通过横向扩展Cluster集群提升性能，相当于提高锁的容量。对于同一个锁的性能的提升，可以使用分段锁。

                                                                                    比如1000库存的商品，每100个商品拆分成一个段，分别放在一个槽位上，那么库存就落在了不通的节点上，加锁只针对于不通的段位加锁即可。

                                                                                    具体实现可以参照CurrentHashMap。
