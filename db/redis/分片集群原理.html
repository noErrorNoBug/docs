                                                                                          # 架构原理

                                                                                          由多个主从节点组成的分片集群，具有复制、高可用和分片特性。分片集群不需要哨兵，也能完成节点移除和故障转移的功能。需要将每个节点设置成集群模式。

                                                                                          集群模式下**没有核心节点，可以水平扩展，**官方不推荐超过1000个。

                                                                                          # 优缺点

                                                                                          • 具有复制、高可用和分片特性
                                                                                          • 没有完全解决访问瞬断的情况，但是不影响其他master节点的访问。
                                                                                          • 高并发能力强

                                                                                          # 数据定位规则

                                                                                          Redis Cluster 使用Hash槽处理实例和数据的对应关系。将所有的数据划分为 16384 个 slot（槽位），每个节点在加入集群后，负责其中一部分槽位。槽位的信息储存于节点中。

                                                                                          Cluster会默认对key使用CRC16算法进行 hash 得到一个数值，然后用这个数值对16384进行取模运算得到具体的槽位。

                                                                                          HASH_SLOT = CRC16(key) mod 16384.

                                                                                          当Redis Client连接集群时，会缓存槽位信息到本地。当客户端需要查找或者set某个key时，会直接定位到目标节点。

                                                                                          # 跳转重定位

                                                                                          同时为了防止client的槽位信息和Cluster不一致，还需要纠正机制。

                                                                                          当客户端向一个错误节点发出指令后，该节点会发现key的槽位不归自己管理，这时它会向客户端发送一个特殊指令（Moved命令），携带目标操作的节点地址，告诉客户端去连接这个地址去获取数据。

                                                                                          客户端收到指令去跳转到正确节点时，也会同步更新本地槽位映射表，后续的key使用新的映射表。

                                                                                          # 数据完整性配置

                                                                                          通过下面配置，可以当一个插槽的主库下线时，即使没有slave进行故障恢复，集群仍可用。

                                                                                          cluster-requier-full-coverage no //no为仍可用，yes为不可用
                                                                                          1

                                                                                          # 批量数据操作

                                                                                          对于类似mset、mget这样多个key操作的原生命令，redis集群只支持把所有的key落在同一个slot的情况，如果多个key一定要用mset在redis上进行操作，可以在key前面加上{xxx}，这样参数数据分片计算hash时只会计算{}中的值，确保同类key落到同一个slot：

                                                                                          mset {user}:1:name zhangsan {user}:1:age 22
                                                                                          1

                                                                                          # 集群节点通讯原理

                                                                                          Redis Cluster 的节点之间使用gossip协议进行通讯，其中维护集群元数据的方式有两种：集中式和gossip。

                                                                                          # 集中式

                                                                                          集中式的优点在于元数据的读取和更新的时效性很好，一旦元数据出现变化就会立即更新到集中式的存储中，其他节点在读取的时候就会立即感知到。

                                                                                          但是集中式的缺点是将压力全部集中在一个地方，可能导致元数据存储压力过大。很多中间件会利用Zookeeper做集中式的元数据存储。

                                                                                          # gossip

                                                                                          gossip协议包含多种消息，如ping、pong、meet、fail等：

                                                                                          • meet：某个节点发送meet消息给新节点，邀请其加入集群，然后新节点开始与其他节点通讯。
                                                                                          • ping：每个节点都会平凡的给其他节点发送ping消息，其中包含自己的状态和维护的元数据，互相通过ping消息维护和更新元数据。
                                                                                          • pong：对ping和meet的消息返回，包括自己的状态和其他信息，也可以用于消息的广播和更新。
                                                                                          • fail：某个节点判断到一个节点fail后，就会发送fail消息给其他节点，通知其他节点那个节点宕机了。 gossip协议对于元数据的更新比较分散，不集中在一个地方，更新请求陆陆续续，会有一定的延迟，降低了压力；缺点在于可能会导致一定的集群操作滞后。

                                                                                          Redis每个节点有一个专门用于gossip通信的端口，一般是10000+正常通信端口，如16379。

                                                                                          # 网络抖动

                                                                                          由于物理网络会出现抖动这种常见现象，为了避免集群判定的误差，可以设置 cluster-node-timeout，表示节点持续多长时间失联后，才判定为故障，进行主从切换。

                                                                                          这样可以避免因为网络抖动造成的频繁切换。

                                                                                          # 选举原理及过程

                                                                                          当slave发现自己的master变为FAIL状态时，便尝试进行Failover，以期待成为新的master。由于挂掉的master可能存在多个slave，多个slave竞争过程如下：

                                                                                          1. slave发现master变为FAIL状态。
                                                                                          2. 将自己记录的集群currentEpoch加1，并且广播FAILOVER_AUTH_REQUEST信息。
                                                                                          3. 其他节点接收到该信息，只有master节点响应，判断请求合法性，并且发送FAILOVER_AUTH_ACK信息，对每一个epoch只发送一次ack。
                                                                                          4. 尝试failover的节点手机master返回的FAILOVER_AUTH_ACK。
                                                                                          5. slave收到超过半数的ack后变为master节点。（这里可以说明为什么最少需要3个master节点，如果两个节点，挂掉后无法选举）
                                                                                          6. slave广播Pong消息通知其他节点自己变成master。 slave并不是在master进入FAIL状态的一瞬间就发起选举，而是有一定的延迟以保证FAIL状态在集群中传播，如果没有延迟，有可能其他Master不知道FAIL状态而拒绝选举。

                                                                                          延迟计算公式如下：

                                                                                          DELAY = 500ms + random(0-500ms) + SLAVE_RANK * 1000ms

                                                                                          • SLAVE_RANK：表示slave已经从master复制数据的总量rank，rank越小代表数据越新，这种方式下，持有最新数据的slave会最先发起选举。

                                                                                          # RedisCluster为什么至少需要3个master节点？并且推荐节点数为奇数？

                                                                                          新master的选举需要超过半数的同意，如果只有两个master的话，如果一个master转入FAIL，将无法进行投票。

                                                                                          奇数个master节点可以满足选举的基础上节省一个节点，比如3个master和4个master，如果大家都挂了一个master，都可以进行选举，如果都挂了2个master，就都无法进行选举，所以奇数个master是从节省资源的角度出发。

                                                                                          # 脑裂数据丢失

                                                                                          脑裂问题主要指的是，当主节点和从节点网络不通，就会发起新一轮的选举，这个时候client可能往老master和新master上都写入了不通的数据，当老的master恢复通信后尝试加入集群时，会变成从节点，就会从新master中同步数据，那么它自身的数据都会丢失。

                                                                                          redis集群没有过半机制会有脑裂问题，网络分区导致脑裂后多个master对外提供写服务，一旦网络恢复，会将其中一个主节点变为从节点，这时会有大量的数据丢失。

                                                                                          规避方法可以添加以下参数：

                                                                                          min-replicas-to-write 1 //写数据成功最少同步的slave数量，这个数量可以模仿大于半数机制配置，比如 集群总共三个节点可以配置1，加上leader就是2，超过了半数
                                                                                          1

                                                                                          这个配置一定程度上影响集群可用性，如果slave少于一个，就算集群leader正常也不能提供对外服务了。
