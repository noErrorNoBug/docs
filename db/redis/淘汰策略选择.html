                                                                              # 缓存容量设置

                                                                              根据二八原理（统计结果），一般情况下，20%的数据提供了80%的访问，剩下的虽然数据量大，但是访问量只是少的可怜。但是在实际访问中，应该根据数据的实际访问特征和成本开销综合考虑。

                                                                              一般情况下或者不知道访问特征的情况下，可以设置为总量的15%-30%左右。

                                                                              同时，缓存被写满是不可避免的，因此要选择合适的淘汰策略。

                                                                              # 缓存淘汰策略

                                                                              Redis在4.0后总共设置了8中缓存淘汰策略，一种不淘汰的策略，4种针对设置过期时间的数据进行淘汰的策略，3中针对所有数据进行淘汰的策略。

                                                                              # 不淘汰策略

                                                                              noeviction策略是不淘汰所有数据，缓存一旦被写满，将停止Redis的服务端服务，返回错误结果，我们一般不用这种淘汰策略。

                                                                              # 过期数据淘汰

                                                                              对于设置过期时间的淘汰策略分别是下面4中：

                                                                              • volatile-ttl：根据过期时间先后进行淘汰，越早过期时间越早淘汰
                                                                              • volatile-randoms：在设置了过期时间的数据中随机淘汰
                                                                              • volatile-lru：根据lru算法筛选设置过期时间的键值对
                                                                              • volatile-lfu：根据lfu算法筛选设置了过期时间的键值对 **LRU：**最近最少使用，淘汰很久没有使用的数据（以访问时间为依据），热点数据使用LRU效果好，因为是时间突发

                                                                              **LFU：**最不经常使用，淘汰访问次数最少的数据（以访问次数为依据），周期性的使用LFU效果好

                                                                              # 所有数据淘汰

                                                                              对于所有键值对的淘汰策略有下面三种：

                                                                              • allkeys-randoms：随机淘汰所有键值对
                                                                              • allkeys-lru：根据lru算法筛选所有键值对
                                                                              • allkeys-lfu：根据lfu算法筛选所有键值对

                                                                              # LRU算法

                                                                              LRU算法全称是最近最少使用算法，是按照最近时间内最少使用次数的原则，把不常用的数据筛选出来，而频繁使用的数据留在内存中。

                                                                              LRU算法是将数据组织成一个链表，链表的一端叫MRU端，表示最常使用的数据，另一端是LRU端，表示最不长使用的数据。

                                                                              刚刚被访问的数据会被移动到MRU端，最不长使用的数据随着常用数据的移动，被排列到LRU端。

                                                                              新加入的数据被放置在MRU端，LRU端的数据如果满了就会被剔除。

                                                                              LRU有个非常朴素的思想认为，刚刚被访问的数据接下来还会被访问，长久不访问的数据之后就不会被访问了。

                                                                              # LFU算法

                                                                              LFU算法是在LRU的基础上，增加了一个次数计数器，每个链表中的元素都有一个访问次数计数器，每次访问次数+1。

                                                                              在做筛选时，会首先根据访问次数进行筛选，访问次数最低的将淘汰，如果访问次数相同，则使用LRU进行淘汰。

                                                                              # 如何处理已经被淘汰的数据

                                                                              一般情况下，我们在使用缓存时，干净的数据会直接删除，脏数据会回写数据库，但是Redis的机制中，淘汰的数据会直接删除，因此我们要在Redis数据发生变化时，还未被删除前，就要从应用端与之与数据库同步。
