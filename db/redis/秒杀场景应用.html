                                                                          # 秒杀场景的特点

                                                                          • 瞬时并发访问量非常高（要求Redis拦截大部分请求，避免压力直接打到数据库）
                                                                          • 读多写少，读操作是简单的查询

                                                                          # Redis在秒杀场景的应用环节

                                                                          # 秒杀活动前

                                                                          **把商品详情页的页面元素静态化，使用CDN或者浏览器把静态元素缓存起来。**大量的秒杀前的请求就不会到达服务端，减小服务端压力。

                                                                          # 秒杀活动开始

                                                                          业务场景主要是三个的循环

                                                                          1. 库存查验：用户点击秒杀后查验库存
                                                                          2. 库存扣减：如果有库存则库存扣减，没有库存返回失败接下来继续点击秒杀回到1
                                                                          3. 订单处理：库存扣减后生成订单和后续操作 由于步骤3是库存扣减后才会进行的后续处理，因此压力主要集中在库存查验的操作上。因此库存查验和库存扣减需要放在Redis上以避免大量的压力，订单处理可以放在数据库中进行。

                                                                          订单处理可以放在数据库的原因如下：订单处理还涉及到商品出库、支付、物流等一系列的其他系统的操作，要保证事务性，因此是可以放在数据库中进行操作的。

                                                                          库存扣减要在Redis中执行的原因：

                                                                          • 数据库额外的开销：既要在数据库中做实际扣减操作，又要维护Redis的库存数据同步，开销非常大。
                                                                          • 下单时超过实际库存出现超售：由于数据库更新操作慢，如果出现双写不一致，就会出现超售的情况。 因此需要在Redis中进行减库存的操作，具体的操作是：当库存查验完成后，如果有余量则在Redis中直接进行扣减，而且要保证库存查验和库存扣减的原子性。

                                                                          # 秒杀活动结束后

                                                                          秒杀结束后一般都是刷新页面或者跟踪订单，用户访问量也会急剧下降，因此场景就少了。

                                                                          # 实现方案

                                                                          秒杀场景的根本操作要求主要有两个：

                                                                          • 高并发：Redis本身支持高并发，还可以使用切片机群，不同的商品分配到不同的切片上，分担压力。
                                                                          • 保证库存查验和库存扣减的原子性：Lua脚本或者分布式锁。

                                                                          # 系统性工程

                                                                          • 页面静态化。秒杀页面上能静态化处理的页面元素，我们都要尽量静态化，这样可以充分利用 CDN 或浏览器缓存服务秒杀开始前的请求。
                                                                          • 请求拦截和流控、请求分段放行。在秒杀系统的接入层，对恶意请求进行拦截，避免对系统的恶意攻击，例如使用黑名单禁止恶意 IP 进行访问。如果 Redis 实例的访问压力过大，为了避免实例崩溃，我们也需要在接入层进行限流，控制进入秒杀系统的请求数量。
                                                                          • 库存信息过期时间处理。Redis 中保存的库存信息其实是数据库的缓存，为了避免缓存击穿问题，我们不要给库存信息设置过期时间。
                                                                          • 数据库订单异常处理。如果数据库没能成功处理订单，可以增加订单重试功能，保证订单最终能被成功处理。
                                                                          • 消息队列处理订单。订单可以在库存扣减成功后发送给消息队列进行异步处理。
